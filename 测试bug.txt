四、问题与优化项

BUG-001 README 和 PRD 的棋盘尺寸描述与当前实现不一致
严重程度：中
类型：文档/验收口径错误
状态：已修复（1.2.8优化）

复现/检查：
1. 查看 README.md 当前游戏范围。
2. 查看 docs/PRD.md 第 4 节关卡与难度规则。
3. 对比 src/levels.js、tests/progression.test.mjs 和 tests/engine.test.mjs。

实际结果：
- 原 README.md 曾写“每个主题内 1-10 关为 6列 x 7行，11-20 关为 6列 x 8行，21-30 关为 6列 x 9行”。
- 原 docs/PRD.md 表格同样曾写 normal 为 6列 x 8行、hard 为 6列 x 9行。
- 当前代码 src/levels.js 中 easy/normal/hard 都是 rows: 7, cols: 6。
- tests/progression.test.mjs 明确断言 “each chapter repeats easy, normal and hard tiers on the easy board layout”。
- 本轮单测和 smoke 都按统一 6列 x 7行口径通过。

期望结果：
README 和 PRD 应同步为当前实现：三档关卡当前统一使用 6列 x 7行棋盘，normal/hard 通过图案种类和时间递减体现难度。

影响：
后续测试人员如果按 README/PRD 验收，会把当前正确实现误报为棋盘尺寸错误，或者反过来漏掉真实布局回归。

处理：
README.md 和 docs/PRD.md 已同步为当前实现：三档关卡统一使用 6列 x 7行棋盘，normal/hard 通过图案种类和时间递减体现难度。

OPT-001 src/engine.js 仍保留旧 demo 关卡常量，和正式 90 关配置不一致
严重程度：低
类型：代码维护/误用风险
状态：已优化（1.2.8优化）

复现/检查：
1. 查看 src/engine.js 顶部 export const LEVELS。
2. 查看 src/levels.js 中正式 90 关 LEVELS。

- 这组旧常量的 hints/shuffles 分别为 3/2、4/3、3/2。
- 当前正式关卡配置在 src/levels.js，easy/normal/hard 的 hints/shuffles 均为 0。
- 运行时 game.js 使用 src/levels.js 的 LEVELS；engine.js 的 LEVELS 已清理，tests/engine.test.mjs 改为从 src/levels.js 读取正式关卡配置。

期望结果：
引擎层不要继续暴露容易和正式关卡配置混淆的 LEVELS 名称，或至少改名为 DEMO_LEVELS / ENGINE_TEST_LEVELS，并注明不作为运行时关卡数据源。

处理：
src/engine.js 已移除旧 demo LEVELS 导出，并新增单测断言 engine 模块不再导出 legacy demo level presets。

OPT-002 浏览器会反复输出 WebAudio 自动播放策略警告
严重程度：低到中
类型：音频体验/真机风险
状态：已优化，待真机复核

复现步骤：
1. 使用浏览器打开首页。
2. 监听 console warning。
3. 进入设置页、返回首页、开始游戏。

实际结果：
在 320/360/390/1440 视口补测中，Chrome 多次输出：
The AudioContext was not allowed to start. It must be resumed (or created) after a user gesture on the page.

期望结果：
理想情况下，页面在用户首次手势前减少主动创建/调度 WebAudio，避免控制台反复警告；或确认当前警告只发生在浏览器自动播放策略下，不影响首次触摸后的 BGM 和音效解锁。

影响：
当前 smoke 已覆盖刷新后首次触摸解锁背景音乐和开局 BGM 逻辑，因此这不是已证实的功能阻塞。但真机 WebView、微信/抖音容器和不同系统浏览器可能对 WebAudio 恢复时机更敏感，需要实机听感验证。

处理：
audio controller 已支持在 suspended AudioContext 下延迟 BGM 调度；首页首次启动音乐会避免在用户手势前主动 resume/schedule suspended BGM，首次触摸/点击后仍走原解锁路径。发布前仍需用真机验证：首页首次触摸空白处、点击设置/个人中心、点击开始闯关后，背景音乐和按钮音效是否按预期响起。

OPT-003 未开放入口对正式用户仍像真实交易入口
严重程度：中
类型：产品体验/发布策略
状态：待产品确认

涉及入口：
1. 首页兑换入口。
2. 道具用完弹框的“购买”。
3. 体力不足弹框的“购买体力”。
4. 失败结果页的“购买道具”。

实际结果：
当前 PRD 和代码口径均为保留入口，点击后统一提示“功能暂未开放”。本轮自动化已覆盖该行为。

期望结果：
如果这是正式测试版，可以接受；如果准备面向真实用户发布，建议明确策略：隐藏未开放入口、置灰禁用并标注未开放，或接入最小闭环。

影响：
玩家会看到一批像广告/购买/兑换的真实入口，但点击后没有资源变化，容易形成“按钮没用”的体验。

建议：
短期发布：弱化为展示态或“敬请期待”。
功能上线：补广告桥、支付/金币消耗、资源变更、失败/取消/重复点击回归测试。
