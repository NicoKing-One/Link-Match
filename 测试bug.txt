四、问题与优化项

BUG-001 PRD、设计说明、README 与当前产品严重不一致
严重程度：中
类型：文档/需求口径错误
状态：未修复

复现/检查：
1. 查看 docs/PRD.md。
2. 查看 docs/design.md。
3. 查看 README.md。
4. 对比当前运行时页面和 docs/project-handoff.md。

实际结果：
- docs/PRD.md 仍写 V1 是“三档难度选择、最佳分、本地按难度保存最高分、无广告/支付、无体力/商城”，还写“没有可用步时系统自动洗牌一次”。
- docs/design.md 仍写结算页展示得分、最佳分、体力入口等旧结构。
- README.md 仍写 V1 只用 localStorage 保存各难度最佳分，且“当前 MVP 没有生成图片文件，图案由 CSS 和字符符号实现”。
- 当前项目实际已经是 90 关章节闯关、体力、金币、星级、个人中心、设置页、双倍金币广告、复活广告、图片资产 UI。

期望结果：
需求文档、设计文档、README 至少要反映当前正式测试版本的核心功能范围，避免后续测试人员按旧需求误报或漏测。

建议：
发布前同步更新 docs/PRD.md、docs/design.md、README.md；旧 MVP 描述可以移到历史记录，不应作为当前验收口径。

BUG-002 project-handoff 局部状态文字不一致
严重程度：低到中
类型：交接文档一致性
状态：未修复

复现/检查：
1. 打开 docs/project-handoff.md。
2. 查看顶部“当前阶段”和第 5 节“当前 Git 状态”。

实际结果：
顶部写当前阶段为 1.2.13 未开放功能文案统一；第 5 节仍写“本轮收口内容：1.2.12刷新后音乐解锁修复”和“本轮提交备注：1.2.12刷新后音乐解锁修复”。

期望结果：
交接文档的当前阶段、最近验证、Git 状态和本轮提交备注应保持一致。

影响：
新线程或测试人员接手时，可能误以为当前代码停在 1.2.12，漏掉 1.2.13 的未开放功能文案验证范围。

建议：
同步第 5 节状态，或把它改成“最近提交/最近收口记录”并明确日期。

OPT-001 多个“功能暂未开放”入口仍以可交易按钮形式出现
严重程度：中
类型：产品体验/发布收口
状态：待确认

涉及入口：
1. 首页兑换入口。
2. 道具用完弹框的“看广告获取”和“购买”。
3. 体力不足弹框的“购买体力”。
4. 失败结果页的“购买道具”。

实际结果：
入口均可点击，并统一提示“功能暂未开放”。当前文案口径已统一，自动化也已覆盖。

期望结果：
如果这是正式测试版，可以接受；如果准备面向真实用户发布，建议明确策略：隐藏未开放入口、置灰禁用并标注未开放，或接入最小闭环。

影响：
玩家会看到一批像广告/购买/兑换的真实入口，但点击后没有资源变化，容易形成“按钮没用”的体验。

建议：
短期发布：保留入口但弱化成“敬请期待/未开放”的展示态。
功能上线：补广告桥、支付/金币消耗、资源变更、失败/取消/重复点击回归测试。

OPT-002 设置页开关刷新后恢复开启，需要确认是否符合发布预期
严重程度：低到中
类型：体验口径
状态：当前代码按会话态设计运行

复现步骤：
1. 进入设置页。
2. 关闭音乐、音效、振动。
3. 刷新页面后再次进入设置页。

实际结果：
刷新后三个开关恢复为“开”。这是当前 audio-settings.js 和 smoke 的既定口径，本轮测试通过。

期望结果：
如果产品希望“每次打开小游戏默认开启”，当前行为正确；如果用户期望关闭后下次仍关闭，则需要改回持久化。

建议：
正式发布前确认产品口径。若保持会话态，后续真机测试重点看背景音乐自动解锁、音量和误触声音是否会让用户反感。

OPT-003 测试截图输出缺少批次隔离，个别文件名不能直接说明状态
严重程度：低
类型：测试流程/证据管理
状态：未修复

复现/检查：
1. 运行 scripts/smoke-browser.mjs。
2. 查看 output/playwright。

实际结果：
同一目录下保存多个流程截图，文件名只描述大类。比如 home-map-mobile.png 是 smoke 多个首页流程跑完后的首页截图，本轮显示体力 57/60，不是“新档初始首页”状态。

期望结果：
截图文件名或目录应能表达场景状态，避免人工复核时误解。

建议：
每轮 smoke 输出到带时间戳/场景名的子目录，或把截图命名为 home-after-return-mobile.png、home-fresh-mobile.png 等更精确的名称。
