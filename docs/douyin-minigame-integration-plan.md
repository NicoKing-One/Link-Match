# 抖音小游戏接入与上架计划

更新时间：2026-07-01

## 1. 目标

将当前 `Link Match 连连看` 从移动端竖屏 H5 单机版本，推进到可在抖音小游戏平台预览、测试、提审并最终上架的版本。

本计划优先解决“能不能作为抖音小游戏跑起来、能不能满足平台提审基础要求、哪些能力需要最小接入”三个问题。普通 UI 微调、头衔系统闭环、二级页正式 icon 替换等功能，建议排在抖音平台最小可预览版本之后。

## 2. 当前项目基础

- 当前阶段：`1.2.8优化` 已完成本地提交并推送到远程 `main`。
- 当前形态：移动端竖屏 H5 单机小游戏。
- 当前运行方式：本地静态服务加载 `src/index.html`、`src/styles.css` 和 `src/*.js`。
- 当前核心系统：90 关配置、三主题首页地图、体力、金币、星级、道具、激励广告占位、WebAudio 音效、localStorage 本地存档、浏览器 smoke 回归。
- 当前风险点：抖音小游戏运行环境不等同于普通浏览器页面，不能假设完整 DOM/CSS/H5 路径可以无改动上架。

## 3. 官方接入要点

以下信息来自抖音开放平台官方文档，实际提审前需要再次按后台最新提示核对。

- 入驻与主体认证：创建小游戏前，需要先完成抖音开放平台账号入驻、主体认证与对公验证。
  来源：[注册与入驻](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/register)
- 创建与基础信息：创建小游戏后，需要完善名称、图标、简介、详细介绍、资质等基础信息，并提交平台审核。
  来源：[创建与信息完善](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/improve-information)
- 开发工具与工程结构：需要使用抖音开发者工具开发、预览、上传。若不依赖 Cocos、Laya、Egret 等引擎，需要将工程打包为小游戏结构，目录至少包含 `project.config.json` 与 `game.json`。
  来源：[开发小游戏](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/develop)
- 必接能力：所有小游戏必须接入侧边栏复访能力；内购游戏需要接入客服能力；若游戏内存在用户自定义输入，需要处理敏感词相关能力。
  来源：[必接能力](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/essential-skills)
- 备案申报：待上线小游戏需要基础信息审核通过、版本审核通过后，平台才会自动进行小游戏备案申报。
  来源：[小游戏备案申报](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/game-filing-application-works)
- 版本提审：开发完成后通过 IDE 上传测试版本，再在后台提交审核；首次上线前通常会有 QA 回归。使用网络能力时，需要配置 `https://` 或 `wss://` 安全域名白名单。
  来源：[版本提审指引](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/examineguide)
- 发布与运营配置：版本审核通过后才能发布；上线前建议配置搜索关键词与分享内容。广告能力可在完成主体资质认证和对公验证后，于商业化/流量主路径申请。
  来源：[发布小游戏](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/release)
- 性能：官方建议关注图片尺寸、透明留白、合图、资源复用和纹理内存。当前项目图片资源较多，接入抖音前需要做一次资源体积与运行内存检查。
  来源：[性能优化](https://developer.open-douyin.com/docs/resource/zh-CN/mini-game/guide/minigame/optimization)

## 4. 推荐路线

推荐采用“最小平台适配优先”的路线：

1. 先完成平台账号、主体认证、创建小游戏，拿到 `appID`。
2. 在项目中新增抖音小游戏适配工程，优先跑通开发者工具预览。
3. 保留当前单机核心玩法和已有 UI，先不重做引擎、不重做关卡、不新增头衔系统。
4. 接入最低必需的平台能力：生命周期、存储适配、侧边栏复访、分享/搜索配置、基础埋点占位。
5. 接入激励视频广告能力，映射到现有“双倍金币”“失败复活”“道具获取”三个广告入口。
6. 完成真机扫码测试、平台自审、版本上传和提审。

不建议第一阶段直接重写为 Cocos 或 Unity。当前项目玩法、UI、测试和资源都已经成型，重写会放大工期和回归风险。除非开发者工具验证后发现当前 H5/DOM 架构无法稳定运行在抖音小游戏环境，再考虑第二方案。

## 5. 技术改造范围

### 5.1 新增平台适配目录

建议新增：

```text
platforms/douyin/
```

初始内容建议包括：

```text
platforms/douyin/project.config.json
platforms/douyin/game.json
platforms/douyin/game.js
platforms/douyin/README.md
```

如果后续需要独立构建产物，可再加入：

```text
platforms/douyin/dist/
platforms/douyin/scripts/
```

### 5.2 平台能力封装

不要在 `src/game.js` 中到处直接调用 `tt.*`。建议新增统一平台层，例如：

```text
src/platform/browser-platform.js
src/platform/douyin-platform.js
src/platform/index.js
```

平台层负责屏蔽差异：

- 启动与生命周期：进入前台、退到后台、关闭、恢复。
- 存储：浏览器 `localStorage` 与抖音小游戏本地存储之间的适配。
- 广告：当前 `window.linkMatchAds.showRewardedAd` 占位迁移为平台统一 `showRewardedAd(placement)`。
- 分享：分享标题、分享图、路径参数。
- 复访：侧边栏复访能力。
- 设备信息：安全区域、屏幕尺寸、像素比、宿主端判断。

### 5.3 第一阶段保留的内容

第一阶段尽量保持以下内容不变：

- 90 关配置与进度规则。
- 三主题地图和关卡解锁逻辑。
- 当前金币、体力、星级、提示、洗牌、复活规则。
- 当前音效原则：原创 WebAudio，不依赖外部版权素材。
- 当前自动化测试口径：单测、资源检查、构建、浏览器 smoke 继续保留。

### 5.4 第一阶段需要调整的内容

- 存档读写从直接依赖 `localStorage` 改为通过平台适配层访问。
- 激励广告从浏览器占位切换为平台适配层，浏览器仍保留 fake/mock 路径，方便本地测试。
- 启动入口从普通 HTML 页面扩展到抖音小游戏入口。
- 资源加载路径需要兼容小游戏包内路径。
- 补充抖音开发者工具需要的配置文件。
- 检查并压缩大图资源，避免小游戏包体、加载时间和内存风险。

## 6. 平台材料清单

需要用户在抖音开放平台或资质侧准备：

- 抖音开放平台账号。
- 主体认证材料。
- 对公验证条件。
- 小游戏名称：建议先确认是否使用 `Link Match 连连看`，还是改成更适合中文搜索的名称。
- 小游戏图标。
- 小游戏简介与详细介绍。
- 软件著作权或平台要求的相关资质材料。
- 如涉及版号或备案要求，按后台提示补充。
- 隐私政策确认。
- 适龄提示分段与说明。
- 搜索关键词。
- 分享标题、分享文案、分享图。
- 审核用截图或录屏素材。

## 7. 第一阶段执行计划

### 阶段 A：平台账号与 appID

负责人：用户

目标：

- 完成抖音开放平台账号入驻。
- 完成主体认证和对公验证。
- 创建小游戏，拿到 `appID`。
- 确认小游戏名称、主体名称、图标、简介的初版方向。

交付物：

- `appID`
- 平台后台基础信息填写状态
- 资质缺口列表

### 阶段 B：兼容性 Spike

负责人：开发

目标：

- 新增抖音小游戏工程壳。
- 使用当前项目构建产物尝试进入抖音开发者工具。
- 验证当前 DOM/CSS/H5 架构能复用多少。
- 记录必须改造项。

验收标准：

- 抖音开发者工具能打开项目。
- 至少能看到启动页面或明确失败原因。
- 输出一份兼容性结论：继续适配当前 H5、局部改造、或需要引擎化重构。

### 阶段 C：最小可预览版本

负责人：开发

目标：

- 跑通抖音小游戏入口。
- 接入平台存储适配。
- 接入生命周期适配。
- 接入侧边栏复访能力。
- 保留浏览器本地开发路径。
- 建立抖音平台 smoke 或手工验收清单。

验收标准：

- 开发者工具预览可进入首页。
- 真机扫码可进入首页。
- 可开始第 1 关并完成一局。
- 退出重进后基础存档仍可用。
- 浏览器版本现有测试仍通过。

### 阶段 D：广告与运营能力

负责人：开发 + 用户

目标：

- 申请并接入广告能力。
- 将激励视频映射到现有三个入口：
  - 双倍金币
  - 失败复活
  - 提示/洗牌道具获取
- 配置分享标题、分享图、搜索关键词。

验收标准：

- 广告完成后正常发奖。
- 广告取消或失败时不发奖。
- 所有奖励路径仍能在 mock 环境测试。
- 分享和搜索配置已在后台补齐。

### 阶段 E：提审准备

负责人：用户 + 开发

目标：

- 完成基础信息审核。
- 完成平台自审自查。
- 准备提审版本说明。
- 上传测试版本。
- 真机多端回归。

验收标准：

- 测试版本可扫码体验。
- 主流程、广告、复访、存档、音效、适龄提示、隐私政策均完成检查。
- 后台可提交审核。

## 8. 建议的第一版提审范围

第一版提审建议保持克制：

- 上架核心玩法：连连看闯关。
- 开放第 1 关起逐关解锁。
- 保留三主题 90 关配置，但按当前进度规则锁定。
- 保留体力、金币、星级、提示、洗牌、复活。
- 兑换商城继续显示“功能暂未开放”，不要在第一版接入未完成商业闭环。
- 头衔系统暂不接入，避免金币消耗、拥有状态、佩戴状态和客服/支付边界提前复杂化。
- 不接内购，只先接广告变现。这样可以降低客服、支付、版号和运营规范压力。

## 9. 风险与决策点

### 9.1 H5 架构兼容风险

当前项目依赖 HTML、CSS、DOM 和 WebAudio。抖音小游戏支持普通小游戏工程和主流引擎，但实际运行环境与浏览器不同。第一阶段必须先做兼容性 Spike，不能直接假设可提审。

决策点：

- 如果当前 H5 可以稳定预览，则继续做轻量平台适配。
- 如果 DOM/CSS 支持受限但逻辑可复用，则保留 `src/engine.js`、`src/levels.js`、`src/progression.js` 等核心逻辑，重写渲染入口。
- 如果运行环境完全不适合当前架构，再评估 Cocos/Laya 重构。

### 9.2 资质与备案风险

小游戏备案、软著、版号、适龄提示、隐私政策等事项会影响上架节奏。这些不是代码可以单独解决的，需要用户在平台后台同步推进。

### 9.3 广告能力风险

当前游戏已有激励广告占位逻辑，但正式抖音广告需要平台开通能力、广告位配置和真机验证。第一版应优先保证广告失败时不误发奖。

### 9.4 性能风险

当前项目图片资源较多。抖音小游戏平台强调图片尺寸、透明留白、资源复用和纹理内存。接入前需要检查首包体积、图片尺寸和运行内存。

## 10. 立即下一步

建议从以下两条线并行开始：

1. 用户侧：完成抖音开放平台注册/入驻/主体认证，创建小游戏并拿到 `appID`。
2. 开发侧：准备 `platforms/douyin/` 适配工程设计，先做兼容性 Spike，目标是让当前版本在抖音开发者工具中出现首屏或输出明确失败原因。

拿到 `appID` 后，下一轮开发任务建议命名为：

```text
1.2.9抖音小游戏接入Spike
```

这一轮只追求“抖音开发者工具可打开、真机扫码可验证、形成后续改造清单”，不要同时推进头衔、商城、内购或新 UI。
