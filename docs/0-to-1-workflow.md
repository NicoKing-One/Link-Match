# 连连看游戏从 0 到 1 可执行流程

## 1. 总结判断

游戏项目不要死按“UI -> PRD -> 开发”或“PRD -> UI -> 开发”推进。更稳的顺序是：

```text
想法 -> 产品简报 -> 核心玩法规则 -> 低保真界面 -> 高保真设计稿 -> PRD v1 -> 前端开发 -> 后端/数据 -> 测试 -> 打包 -> 平台审核发布 -> 运营迭代
```

原因：游戏不是普通页面，最先要定的是“怎么玩、怎么赢、怎么失败、怎么重开、怎么计分”。UI 可以早做，但不能在玩法规则还模糊时直接进入开发。

## 2. 标准流程

| 阶段 | 你要产出的东西 | 你可以自己做 | 用插件/我来做会更快 |
|---|---|---|---|
| 1. 想法整理 | 一句话定位、目标用户、核心玩法、胜负条件 | 写一句“我要做什么游戏” | 我帮你整理成产品简报 |
| 2. 玩法规则 | 棋盘尺寸、图案数量、连线规则、提示、洗牌、倒计时、关卡 | 口述规则 | 我帮你补成完整规则文档 |
| 3. 信息架构 | 首页、关卡页、游戏页、结算页、设置页 | 手画草图 | Product Design / Figma / Canva 辅助生成流程图和页面结构 |
| 4. 低保真 UI | 黑白线框图，先看布局不看美术 | 纸笔、白板 | Product Design 生成多个方向供你选 |
| 5. 高保真设计稿 | 颜色、按钮、图标、卡牌、动效状态 | Figma / Pixso 手动设计 | Figma、Canva、ImageGen；本项目图片必须用 `gpt-image-2` |
| 6. PRD 文档 | 功能清单、页面说明、交互规则、验收标准 | Markdown / Word | Documents 插件或我直接生成 `docs/PRD.md` |
| 7. 前端开发 | 可运行游戏 | 自己写代码 | 我根据 PRD + 设计稿实现 |
| 8. 后端数据 | 存档、排行榜、用户、埋点、广告/支付回调 | 平台后台配置 | 我写 API、数据结构、云函数/服务端代码 |
| 9. 测试 | 自动化测试 + 真机测试清单 | 手动玩 | Browser / Playwright / 平台开发者工具 |
| 10. 打包上线 | H5 包、小程序/小游戏包 | 开发者工具上传 | 我准备构建脚本和发布清单 |
| 11. 平台发布 | 微信/抖音审核、灰度、全量 | 你操作账号和资质 | 我给逐项发布 checklist 和驳回修复方案 |

## 3. 关键决策

### 3.1 先 UI 还是先 PRD

推荐顺序：

```text
产品简报 -> 低保真设计 -> PRD v1 -> 高保真设计 -> PRD v2 冻结
```

不要一开始写几十页 PRD，也不要只有漂亮 UI 就开发。

### 3.2 想法怎么变成界面

把想法拆成：

- 用户目标。
- 核心循环。
- 页面列表。
- 每页按钮。
- 每个状态。

连连看最少需要：首页、模式选择、游戏页、暂停页、结算页、设置页。

### 3.3 页面怎么变成前端能看懂的 UI 设计稿

最好用 Figma：

- 每个页面一个 Frame。
- 按钮、卡牌、弹窗、HUD 做成组件。
- 标注颜色、字号、间距、圆角、状态。
- 导出图片资源或提供 Dev Mode 规格。

### 3.4 有设计稿和 PRD 后怎么沟通开发

给我这 5 样最稳：

- `docs/PRD.md`
- 设计稿链接或截图
- 素材目录
- 目标平台
- 技术栈或运行方式

示例话术：

```text
请读取 D:\工作文件\游戏\连连看\docs\PRD.md 和设计稿，
先检查项目结构，然后实现 MVP。
优先完成可玩游戏页、连线判定、计时、重开、结算，
不要加未写进 PRD 的功能。
```

### 3.5 开发前先做什么

- 初始化 Git 仓库。
- 创建 `README.md`、`docs/PRD.md`、`docs/design.md`、`assets/`、`src/`。
- 确定技术栈和目标平台。
- 建第一版任务清单。
- MVP 先做单机可玩，不急着做登录、排行榜、支付。

## 4. 测试、打包和平台发布

### 4.1 测试

分三层：

- 规则单测：连线、阻挡、洗牌、可用步。
- 浏览器自动化测试：开始游戏、错误提示、暂停、主页、布局。
- 真机测试：触控手感、屏幕适配、性能、音效、弱网/前后台。

### 4.2 打包

先打 H5/Web 预览包，确认 MVP；再按目标平台打小游戏包。发布前检查：

- 版本号。
- 合法域名。
- 隐私政策。
- 测试账号。
- 截图、图标、名称、简介。
- 无调试日志。
- 无白屏。
- 无违规引导。

### 4.3 微信小游戏

流程：

```text
安装微信开发者工具 -> 导入小游戏工程 -> 模拟器/调试器测试 -> 预览二维码真机测试 -> 上传体验版 -> 微信公众平台提交审核 -> 审核通过后手动发布
```

### 4.4 抖音小游戏

流程：

```text
注册入驻 -> 创建小游戏 -> 完善基础信息/资质/备案 -> 开发者工具调试 -> 上传测试版本 -> 扫码测试 -> 提交审核 -> 审核通过后灰度或全量发布
```

## 5. 推荐 MVP 范围

第一版只做：

- 移动端竖屏。
- 单机连连看。
- 计时。
- 提示。
- 洗牌。
- 暂停。
- 结算。
- 三关难度。
- 本地最高分。

第二版再做：

- 登录。
- 云存档。
- 排行榜。
- 每日挑战。
- 广告奖励。
- 皮肤系统。

当前项目已经完成第一版单机 MVP，下一步应优先试玩、调参、整理美术方向，而不是立刻接后端。
